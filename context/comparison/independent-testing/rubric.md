# Rubric

Каждый кандидат получает оценки по 100-балльной шкале.

## Удобство API

- 90-100: сценарий выражается напрямую, мало glue-кода, defaults безопасны и понятны.
- 70-89: сценарий выражается без хака, но требует заметной ручной сборки.
- 50-69: рабочее решение возможно, но часть инвариантов приходится проектировать пользователю.
- 20-49: сценарий требует тяжелого custom harness или выглядит неестественно для библиотеки.
- 0-19: сценарий практически не поддерживается.

## Качество документации

- 90-100: нужный сценарий явно описан, есть актуальные примеры и contract/failure semantics.
- 70-89: документация покрывает API, но сценарий нужно собирать из нескольких мест.
- 50-69: базовые сведения есть, но есть пробелы в edge cases.
- 20-49: документация фрагментарна или устаревшая.
- 0-19: без исходников или проб и ошибок сценарий непонятен.

## Качество библиотеки в целом

- 90-100: зрелые abstractions, надежные failure modes, ограничение ресурсов, понятная maintenance story.
- 70-89: зрелая библиотека с ограничениями в scope или ergonomics.
- 50-69: пригодна для части задач, но требует дисциплины пользователя.
- 20-49: существенные риски по надежности, поддержке или переносимости.
- 0-19: неприменима для сценария или не дает нужных гарантий.

## Объем кода

Считать nonblank noncomment lines of code для core implementation сценария на библиотеке. Не считать imports,
package declaration, fixture program, общий тестовый harness и Markdown fencing. Если библиотека unsupported, LOC = 0 и
добавляется причина.

## Итог по сценарию

Сценарный итог для кандидата:

`scenario score = round(0.4 * API + 0.2 * docs + 0.4 * library quality)`.

Отчет должен отдельно указать LOC, потому что он не входит в score автоматически: большой объем кода может быть
оправдан, но должен быть виден.
