# Engineering quality charter

## Статус

Этот документ задает обязательные требования ко всему проекту iCLI. Если возникает конфликт между скоростью
и качеством, выбирается качество.

## Главный стандарт

iCLI должен разрабатываться без спешки, без экономии на проектировании, тестах, документации и API. Цель — не быстро
получить работающий prototype, а построить надежную библиотеку с чистой архитектурой и высоким доверием к поведению.

Любое изменение должно улучшать или сохранять:

- осмысленность проекта и отсутствие бюрократического шума;
- ясность public API;
- изоляцию инвариантов;
- надежность runtime;
- проверяемость поведения;
- качество документации;
- будущую сопровождаемость.

Если изменение делает код быстрее написать, но сложнее понять, проверить или расширить, оно не проходит стандарт
качества.

Если действие или документ не имеет понятного адресата, практической пользы и связи с текущей стадией проекта, оно не
проходит стандарт качества даже при аккуратном исполнении. Формальная полнота не является ценностью сама по себе.

## Обязательные требования к API

- API проектируется от пользовательского сценария, а не от низкоуровневых flags.
- Public surface должен быть маленьким, последовательным и предсказуемым.
- Широкие возможности появляются через композицию value objects и policies.
- Ошибки API должны быть обнаружены как можно раньше: в builder, value object или resolver.
- Public API не должен протекать implementation details, transport quirks или concurrency machinery.
- Каждый public example должен компилироваться, когда появится build infrastructure.

## Обязательные требования к архитектуре

- Все сценарии используют общий execution kernel.
- Каждый инвариант имеет одного владельца.
- Runtime получает нормализованный `ExecutionPlan`, а не набор полусырых builder flags.
- Transport-specific logic изолируется за узкими ports/SPI.
- Сложная возможность сначала доказывается как internal design, затем становится public API.
- Новые public classes добавляются только если они снижают реальную сложность для пользователя.

## Обязательные требования к коду

- Код, комментарии, Javadoc/KDoc, tests и commit messages пишутся на английском.
- Код должен быть простым, но не примитивным: явная модель важнее короткого хака.
- Предпочтительны immutable objects, records, sealed policy families и узкие interfaces.
- Builders остаются draft layer и не содержат runtime orchestration.
- Никаких half-valid public objects вне builder/draft слоя.
- Никаких boolean soup, если за boolean скрывается доменная политика.
- Исключения должны нести structured data, когда это важно для диагностики.

## Обязательные требования к тестам

- Поведение сначала фиксируется тестом или eval-сценарием.
- Каждый важный инвариант имеет проверку.
- Fixture/eval suite — часть продукта, а не вспомогательная мелочь.
- Тесты должны покрывать happy path, failure path и edge cases.
- Timeout, cancellation, stream draining и lifecycle behavior нельзя считать готовыми без негативных тестов.
- Public API examples должны стать compile-tested examples.

## Обязательные требования к документации

- Документация в `context/` пишется на русском.
- У каждого документа должен быть адресат и причина существования: пользовательское действие, maintainer decision,
  архитектурная граница, release gate или проверяемый historical lesson.
- Пользовательская документация должна описывать только реализованное и проверенное поведение.
- Спекулятивные решения живут в ADR или планах, а не в README как обещание.
- Документ должен уменьшать неопределенность. Если он просто повторяет правило, лучше сделать test или validator.
- Документ без реального читателя или release value нужно удалять или сжимать. Нельзя улучшать
  бессмысленный документ только ради полноты.
- Документация обновляется вместе с изменением поведения или API.
- Исторические raw reports, закрытые планы, transcripts и одноразовые audit/subagent outputs не хранятся в `context/`;
  полезные выводы должны быть перенесены в ADR, tests, release/quality docs или текущую документацию.

## Обязательные требования к процессу

- Не спешить ради видимого прогресса.
- Не начинать крупную реализацию без ясного invariant boundary.
- Не добавлять abstraction без доказанной пользы.
- Не добавлять код или документацию без текущей проверяемой пользы.
- Не скрывать сомнения: если решение неочевидно, оформить ADR.
- Не оставлять worktree в неопределенном состоянии после завершения этапа.
- Не выполнять процессный шаг по инерции. Перед действием должно быть понятно, какую неопределенность оно снимает или
  какой риск уменьшает.
- Не накапливать исторический контекст. Git history является архивом проекта.

## Критерии "не готово"

Изменение не считается готовым, если:

- API выглядит удобным только для автора runtime, а не для пользователя;
- правило существует только в тексте, но не в коде или тесте;
- runtime принимает невозможную комбинацию options;
- ошибка теряет важный контекст;
- тест покрывает только happy path;
- документация обещает больше, чем реализовано;
- документ существует без понятной аудитории, решения или практического применения;
- audit finding предлагает довести до полноты документ, полезность которого не доказана;
- новая сущность дублирует уже существующий сценарий или policy;
- агент не может понять источник истины для принятого решения.

## Практическое правило

Лучше меньше возможностей, но с ясными инвариантами и надежными тестами, чем широкая поверхность API, которую
невозможно уверенно сопровождать.

Максимальная версия iCLI должна вырасти из маленького надежного ядра, а не из набора поспешных компромиссов.
