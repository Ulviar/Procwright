# Модель разработки

## Базовый принцип

Используем сильные стороны agent-assisted разработки, но без обязательной бюрократии ради бюрократии.

Агент должен иметь доступ к источникам правды, проверкам и временному плану, если он действительно нужен. Но каждый шаг
не должен требовать полного task dossier.

Обязательный стандарт качества описан в [quality/engineering-charter.md](quality/engineering-charter.md). Скорость
не является самостоятельной целью: решения должны быть надежными, проверяемыми и хорошо документированными.

## Уровни процесса

### Малое изменение

Примеры: исправить тест, добавить маленький метод, поправить формулировку.

Требования:

- прочитать релевантный контекст;
- внести изменение;
- запустить релевантную проверку;
- кратко описать результат.

### Обычная фича

Примеры: добавить capture policy, расширить result, добавить fixture case.

Требования:

- короткий план, если работа неоднозначна;
- тест сначала или одновременно с реализацией;
- обновление scorecard, если меняется статус готовности.

### Архитектурное решение

Примеры: выбрать Java baseline, решить PTY packaging, изменить public API.

Требования:

- ADR в `context/decisions/`;
- явные альтернативы;
- последствия;
- проверка, которая подтвердит решение.

### Рискованное действие

Примеры: удалить большую часть дерева, переписать историю, поменять лицензию, публиковать артефакты.

Требования:

- явное подтверждение пользователя;
- описание области действия;
- план восстановления или честное указание, что восстановления нет.

### Runtime/API-wide изменение

Примеры: изменить lifecycle процесса, добавить новый public scenario, поменять result contract, расширить diagnostics
или затронуть общий execution kernel.

Требования:

- явная граница инварианта;
- тест или eval для happy path, failure path и edge case;
- аудит Integrity and Philosophy и Architecture and Code Quality;
- аудит Documentation and Release Maturity, если меняются контракты, README или context docs.

### Release-релевантное изменение

Примеры: изменить dependency policy, release checklist, compatibility/versioning policy, public API surface или набор
публикуемых модулей.

Требования:

- обновить релевантные документы в `context/release/`;
- запустить все три постоянные роли аудита;
- выполнить финальные release gates для затронутой области.

## Контекстная гигиена

- Не читать весь репозиторий по привычке.
- Читать только документы, релевантные текущему файлу или решению.
- Не копировать инструкции, если у них нет текущей пользы.
- Если правило повторяется, лучше сделать тест или script.
- Не коммитить исторические raw reports, закрытые планы, transcripts, одноразовые audit/subagent outputs или benchmark
  results в `context/`.
- Если исторический материал содержит полезный вывод, перенести вывод в ADR, quality/release doc, test или validator, а
  исходный материал удалить. Git history является архивом.

## Планирование

План нужен, когда:

- есть несколько реалистичных вариантов;
- работа меняет public API;
- работа затрагивает несколько модулей;
- действие сложно откатить;
- задача может выйти за одну сессию.

План не нужен для очевидного локального изменения.

Если план был создан для длинной работы, после завершения его нужно удалить или сжать результат в текущий источник
истины. Постоянная папка completed plans не используется.

## Проверки

Для docs-only шагов обязательны ручная проверка структуры Markdown, consistency с `context/README.md` и релевантный
документационный аудит. Для кодовых шагов минимальная проверка:

- compile;
- unit tests;
- fixture/eval scenarios для затронутого поведения;
- formatting;
- public API examples compile.

## Работа с агентами

Принципы из `agents-best-practices`, примененные к этой ветке:

- модель предлагает, репозиторий проверяет;
- рискованные side effects требуют подтверждения;
- контекст должен быть коротким и релевантным;
- долговременная память живет в документах, ADR и tests;
- повторяющиеся ошибки превращаются в проверки;
- subagents не нужны до появления измеримой проблемы.
