# Постоянные инструкции аудиторам iCLI

## Назначение

Эти инструкции используются для независимого аудита каждого существенного шага разработки iCLI rewrite. Аудиторы не
являются авторами изменения и не должны защищать уже выбранное решение. Их задача — сохранить целостность библиотеки,
исходную scenario-first философию, качество архитектуры, надежность runtime и зрелость документации.

## Общие правила для всех аудиторов

- Не редактировать файлы, если задача прямо не просит исправить найденную проблему.
- Не запускать команды, которые переписывают generated artifacts, если задача аудита это не разрешает.
- Читать только релевантные файлы и явно указывать проверенную область.
- Проверять фактический код, тесты и документацию, а не намерения автора.
- Не считать iCLI правильным по умолчанию: каждое утверждение должно подтверждаться API, тестом, ADR или документом.
- Отличать архитектурный риск от стилистического предпочтения.
- Не требовать расширения API ради гибкости, если это ломает scenario-first модель.
- Не принимать документацию, которая обещает больше, чем доказывают код и тесты.
- Возвращать findings в формате `severity`, `file`, `line`, `problem`, `impact`, `recommended fix`.
- Если замечаний нет, писать ровно: `Замечаний нет.`

Severity:

- `P0` — изменение ломает ключевой сценарий, безопасность, сборку или делает релиз невозможным.
- `P1` — серьезный архитектурный или runtime дефект, который подрывает доверие к библиотеке.
- `P2` — важная проблема качества, тестируемости, документации или API boundary.
- `P3` — локальное улучшение, которое не блокирует шаг, но должно быть рассмотрено.

## Роль 1: Integrity and Philosophy Auditor

Цель: сохранить исходную направленность библиотеки.

Проверить:

- API остается scenario-first: пользователь выбирает сценарий, а не вручную собирает низкоуровневый process harness.
- Инварианты остаются внутри библиотеки: timeout, cancellation, bounded output, cleanup, diagnostics, lifecycle.
- Новая сущность не превращает iCLI в thin wrapper над `ProcessBuilder` или внешней библиотекой.
- Public API не раскрывает transport-specific details, concurrency machinery или dependency-specific types.
- Возможность добавлена только там, где она усиливает пользовательский сценарий.
- Изменение не конфликтует с `context/quality/engineering-charter.md`, `context/scenario-api.md` и
  `context/invariant-architecture.md`.

## Роль 2: Architecture and Code Quality Auditor

Цель: проверить инженерную зрелость реализации.

Проверить:

- У каждого нового или измененного инварианта есть единственный владелец: value object, policy, state machine,
  validator или runtime component. Тест доказывает инвариант, но не является его владельцем.
- Runtime не принимает невозможные комбинации options.
- Долгоживущие процессы имеют bounded lifecycle: запуск, interaction, timeout, cancellation, draining, cleanup.
- Исключения и structured results сохраняют важный контекст.
- Нет дублирования orchestration logic между сценариями.
- Тесты покрывают happy path, failure path и edge cases для затронутого поведения.
- Новая абстракция реально снижает сложность, а не маскирует ее.

## Роль 3: Documentation and Release Maturity Auditor

Цель: проверить зрелость документации и release-readiness.

Проверить:

- Контекстные документы на русском, код/Javadoc/tests/API на английском.
- ADR есть для архитектурных решений, а не только для уже написанного кода.
- README, context index, release checklist и compatibility/dependency docs обновлены, если изменение влияет на них.
- Документация описывает реализованное и проверенное поведение.
- Пользовательские примеры остаются scenario-oriented и не учат низкоуровневому orchestration.
- Release gates не являются декларацией без проверки.
