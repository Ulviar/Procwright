# Уроки старой реализации

## Что переносим

### Продуктовая идея

iCLI должен решать реальную проблему: надежное управление внешними CLI-процессами из JVM-кода. Сохраняем фокус
на one-shot командах, интерактивных sessions, line-oriented workflows и prompt automation.

### API-эргономика

Старый README хорошо нащупал пользовательскую форму:

- базовая команда конфигурируется один раз;
- отдельный запуск настраивается lambda builder;
- one-shot и session flows выглядят родственно;
- результат структурирован;
- сложные настройки доступны, но не обязательны.

### Process safety

Сохраняем как обязательные требования:

- concurrent stdout/stderr draining;
- bounded capture;
- explicit charset;
- direct argv by default;
- explicit shell mode;
- timeout and cancellation;
- graceful shutdown before hard kill;
- optional PTY.

### Детерминированный fixture

Старый `process-fixture` был правильным направлением. В новой ветке fixture/eval suite должен стать главным
способом доказать поведение ядра.

### Agent-legible контекст

Сохраняем идею, что знания должны жить в репозитории, а не в чате. Но новая версия должна быть легче: короткие
документы, ADR, scorecard, evals.

## Что не переносим автоматически

### Process pooling

Pooling был реализован слишком рано и создал слишком много публичных концепций. В clean rewrite его можно возвращать
только поэтапно: сначала узкий `PooledLineSession` поверх уже доказанного `LineSession`, без отдельного runtime,
affinity и lease scope. Более широкий pooling можно добавлять позже, когда будут доказаны:

- стабильная session abstraction;
- реальные warm-worker сценарии;
- reset semantics;
- stress tests.

### Essential/Advanced split

Идея "простое поверх сложного" полезна, но слова `Essential` и `Advanced` не должны становиться публичной
архитектурной рамкой.

### Сценарная таксономия как class design

Каталог сценариев полезен для тестов и документации, но не должен автоматически создавать отдельные public classes.
Сначала primitives, затем helpers.

### Большая workflow-бюрократия

Старые task dossiers помогали агентам, но стали слишком тяжелыми. В rewrite используем risk-tiered процесс:

- простое изменение — без dossier;
- архитектурное решение — ADR;
- длинная работа — короткий активный план;
- проверяемое ожидание — eval/test.

### Документы, обещающие будущий код

README и публичные docs должны описывать только реально работающее поведение. Спекулятивные решения живут в ADR
или plans.

## Главная коррекция

Старая версия пыталась заранее построить полную платформу. Новая должна сначала построить маленькое ядро, которое
трудно сломать.
